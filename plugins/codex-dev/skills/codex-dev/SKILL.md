# codex-dev

Claude는 **기획자 겸 검수자**, 코딩 백엔드는 **구현자** 역할을 맡는 워크플로우.
Claude가 직접 Edit/Write로 코드를 작성하지 않고, 아래 절차로 위임 → 검수 → 재요청을 반복한다.

## 위임할지 직접 짤지 판단 (게이트)

위임 왕복에도 비용이 든다 — 프롬프트 작성 + 검수 + 재요청을 합치면, 작은 작업은 Claude가
직접 짜는 것과 비슷하거나 더 비싸다. 절감 효과는 코드 생성 토큰이 압도적인 큰 작업에서만
실재한다. 그래서 이 스킬이 호출됐더라도 **가장 먼저**(아래 "권한 자동 허용"·"모델·이펙트 선택"
단계보다 먼저) 작업 크기를 판단한다:

**Claude 직접 구현 (이 워크플로우를 쓰지 않음):**
- 파일 1~3개 수준의 수정, 버그 수정, 작은 리팩터링
- 대화 맥락·정밀함이 중요한 코드 (실험·연구 코드, 미묘한 로직)
- 명세를 한 번에 다 적기 어려워 왕복이 많이 예상되는 작업

**Codex 위임 (이 워크플로우 진행):**
- 크고(대략 파일 4개 이상 또는 수백 줄 이상) 기계적이며 명세를 한 번에 적을 수 있는 작업
- 예: 보일러플레이트, 스캐폴딩, CRUD/UI 페이지, 마이그레이션, 반복 패턴 일괄 적용

직접 구현 쪽이면 사용자에게 한 줄로 알리고("작은 작업이라 위임 없이 직접 구현합니다") 평소처럼
직접 구현한다 — 권한 세팅·모델 선택 질문·태스크 생성은 하지 않는다. 단, 사용자가 명시적으로
위임을 요구하면 판단과 무관하게 위임한다.

## 코딩 백엔드 (교체 가능 지점)

현재 백엔드: **Codex** (`mcp__codex__codex`, `mcp__codex__codex-reply` MCP 도구)

> 나중에 다른 코딩 백엔드로 바꾸고 싶으면 이 섹션과 "2. 구현 위임"·"4. 자동 재요청" 단계의
> 도구 이름·파라미터만 교체하면 된다. 나머지 절차(계획 → 위임 → 검수 → 재요청 → 보고)는 그대로 유지.

## 최초 실행 시 Codex 권한 자동 허용

`mcp__codex__codex` / `mcp__codex__codex-reply` 호출마다 매번 승인 팝업이 뜨지 않도록, 이 스킬은
스스로 권한을 세팅한다 (사용자가 `settings.json`을 따로 손대지 않아도 됨. 아래 "모델·이펙트 선택"
질문보다도 먼저 확인한다):

- `~/.claude/settings.json` 을 Read로 확인한다. `permissions.allow` 배열에 `mcp__codex__codex`,
  `mcp__codex__codex-reply` 가 이미 있거나 `permissions.defaultMode` 가
  `bypassPermissions`/`acceptEdits`/`dontAsk` 중 하나면 이미 충분하니 그대로 둔다.
- 둘 다 아니면, 기존 설정을 보존한 채(다른 allow 규칙이나 필드를 지우지 않고) 이 두 값을
  `permissions.allow` 배열에 추가해 병합한다. 파일이 없으면 `{}` 로 새로 만든 뒤 추가한다.
  수정 전 반드시 현재 내용을 Read로 확인하고 병합하며, 통째로 덮어쓰지 않는다.
- 이렇게 하면 이 SKILL.md 파일 하나만 다른 머신/사용자 환경에 복사해도, codex-dev를 처음
  실행하는 순간 자동으로 Codex 호출 권한이 확보된다 — 별도 설치 단계가 필요 없다.

## 최초 실행 시 Codex 모델·이펙트 선택

이 스킬이 시작되면 `TaskCreate`보다도 먼저, 딱 한 번 `AskUserQuestion`으로 이번 실행에서 쓸
Codex 모델과 reasoning effort를 사용자에게 묻는다. 한 세션 안의 모든 `mcp__codex__codex` /
`mcp__codex__codex-reply` 호출에 동일하게 적용한다.

- **모델**: 옵션에 최소 "기본값 유지 (`~/.codex/config.toml`의 현재 모델)"를 recommended로 포함하고,
  사용자가 다른 모델명을 자유롭게 적을 수 있도록 "다른 모델 사용 (직접 입력)" 옵션도 둔다
  (AskUserQuestion의 기본 "Other" 입력으로 커버됨). 선택 결과를 이후 모든 `mcp__codex__codex`
  호출의 `model` 파라미터로 전달한다 (기본값을 골랐으면 `model` 파라미터 자체를 생략해
  config.toml 기본값을 그대로 쓴다).
- **이펙트(reasoning effort)**: `low` / `medium` / `high` / `xhigh` 4개 중 하나를 고르게 한다
  (기본값은 config.toml의 현재 `model_reasoning_effort` 값을 recommended로 표시). 선택 결과는
  `mcp__codex__codex` 호출의 `config: { model_reasoning_effort: "<선택값>" }` 로 전달한다.
  `mcp__codex__codex-reply`에는 이 파라미터가 없으므로, 같은 threadId로 이어가는 이상 최초 호출
  때 지정한 effort가 세션 내내 유지된다 (매 reply마다 다시 지정할 필요 없음).

선택된 모델/이펙트는 태스크 설명이나 메타데이터에 기록해두고, "2. 구현 위임" 단계의 모든
`mcp__codex__codex` 호출에 일관되게 적용한다.

## 작업 진행 상황 표시 (Task 목록)

이 스킬을 시작하면 **가장 먼저** `TaskCreate`로 아래 4단계를 태스크로 생성한다
(사용자가 어느 단계가 진행 중인지 항상 볼 수 있게):

1. `계획 수립: <작업명>`
2. `Codex 구현: <작업명>`
3. `Claude 검수: <작업명>`
4. `마무리 보고: <작업명>`

태스크 목록은 계층 구조를 지원하지 않으므로, 구현을 여러 부분으로 나눌 때는
`Codex 구현` 태스크 하나 대신 `Codex 구현: 로그인 API`, `Codex 구현: 프론트 폼` 처럼
**부분별로 나란히 여러 개** 만든다. 생성 직후 `TaskUpdate`의 `addBlockedBy`로
구현 → 검수 → 보고 순서의 의존 관계를 걸어두면 목록에서 진행 순서가 그대로 보인다.

각 단계에 들어갈 때 `TaskUpdate`로 해당 태스크를 `in_progress`로, 끝나면 `completed`로 바꾼다.
검수에서 문제를 발견해 재요청하는 경우 해당 `Codex 구현` 태스크를 다시 `in_progress`로
되돌렸다가, 수정이 끝나면 `completed`로 바꾸고 검수를 이어간다.

## 절차

### 0. 계획 수립
- `TaskCreate`로 위 4단계 태스크를 만들고 `계획 수립`을 `in_progress`로 표시한다.
- 사용자 요청을 분석해 **무엇을 만들지, 어떤 파일을 건드릴지, 완료 기준이 무엇인지** 명확히 정리한다.
- 애매한 요구사항이면 구현 위임 전에 사용자에게 먼저 확인한다 (AskUserQuestion).
- 기존 코드 구조를 파악해야 하면 Explore 에이전트나 Grep/Read로 직접 조사한다 (이 단계는 Claude가 수행).
- **검수 기준선을 기록한다**: git 저장소면 `git status`로 위임 전 상태를 확인해둔다
  (커밋되지 않은 기존 변경이 있으면 나중에 diff에 섞이므로, 사용자에게 먼저 커밋/스태시할지 확인).
  git이 아니면 건드릴 파일 목록을 기록해두고 검수 때 그 파일들을 Read로 비교한다.
- 계획이 정리되면 `계획 수립`을 `completed`로 표시한다.

### 1. 구현 프롬프트 작성
- 작업을 어떻게 위임할지 형태를 정한다:
  - **병렬 분할** — 서로 독립적인 부분(건드리는 파일이 겹치지 않는 모듈/기능)으로 나뉘면,
    `Codex 구현` 태스크를 부분별로 여러 개 만들고 (예: `Codex 구현: 로그인 API`)
    각각 별도의 Codex 세션을 병렬로 띄운다 (아래 2번 "병렬 세션" 참고).
  - **단계별 순차 위임** — 규모가 있는 작업(예상 변경이 파일 2개 이상 또는 약 100줄 이상)은
    한 번에 통째로 맡기지 말고, 순서 있는 단계로 쪼갠다 (예: 1단계 데이터 모델 → 2단계 API →
    3단계 연결). **한 세션(threadId 하나) 안에서** 단계마다 위임 → 즉시 검수 → 피드백과 함께
    다음 단계 지시(reply)를 반복한다. 잘못된 방향을 조기에 잡는 준실시간 검수 방식 (아래 2번
    "단계별 순차 위임" 참고). 병렬 분할과 조합 가능 (독립 부분끼리 병렬, 각 부분 안에서 단계별).
  - **한 번에 위임** — 위 둘에 해당하지 않는 작은 작업은 단일 세션에 통째로 맡긴다.
    억지로 나누지 않는다 — 나눴을 때 공유해야 할 맥락이 더 많아지면 나누지 않는 게 낫다.
- 각 세션에 보낼 프롬프트에는 다음을 포함한다:
  - 목표(무엇을, 왜)
  - 관련 파일 경로 / 디렉토리 구조
  - 지켜야 할 제약(기존 스타일, 사용 중인 라이브러리, 하지 말아야 할 것, 다른 병렬 작업과 겹치지 않는 범위)
  - 완료 기준(테스트 통과, 특정 동작 확인 등)
- 모호하게 위임하지 말고, Claude가 직접 코드를 짤 때 쓸 법한 구체적인 지시를 그대로 전달한다.

### 2. 구현 위임
해당 `Codex 구현` 태스크를 `in_progress`로 표시한 뒤 `mcp__codex__codex` 호출:
- `prompt`: 위 1단계에서 작성한 지시
- `cwd`: 현재 프로젝트 루트 (절대경로)
- `sandbox`: `danger-full-access` (Codex에 모든 권한 허용 — 항상 이 값 사용)
- `approval-policy`: `never` (승인 요청 없이 자동 진행 — 항상 이 값 사용)
- 응답에서 **threadId를 반드시 기록**해둔다 (재요청에 필요. 여러 세션이면 태스크별로 threadId를 매핑해둔다).

**병렬 세션**: 독립적인 부분이 여러 개면 `mcp__codex__codex` 호출 여러 개를 한 메시지에서 병렬로 보낸다
(각 호출 = 하나의 `Codex 구현` 태스크 = 하나의 독립된 Codex 세션). 서로 다른 파일/모듈을 건드리게 범위를 나눠
충돌을 피한다. 모든 세션이 끝나면 각 태스크를 `completed`로 표시하고 3단계로 넘어간다.

**단계별 순차 위임** (1단계에서 이 방식을 택한 경우):
- 첫 `mcp__codex__codex` 호출의 prompt에 **전체 계획(모든 단계 개요)과 1단계의 상세 지시**를 함께 담는다
  — Codex가 전체 그림을 알아야 1단계를 확장 가능하게 짠다. "1단계까지만 구현하고 멈춰라"를 명시한다.
- 단계가 끝날 때마다 Claude가 **즉시 그 단계 변경분을 검수**한다 (3단계 기준 적용, 단 그 단계 범위만).
  - 문제 없음 → 같은 threadId로 `mcp__codex__codex-reply`: "1단계 확인 완료. 2단계 진행: <상세 지시>"
  - 문제 있음 → 같은 reply에 **수정 지시와 다음 단계 지시를 함께** 담아 왕복을 아낀다
    ("X를 이렇게 고치고, 그다음 2단계 진행: ...").
- 태스크 표시: 단계가 3개 이상이면 `Codex 구현 (1/3): 데이터 모델` 처럼 단계별 태스크로 만들어
  진행률이 보이게 한다. 2단계짜리면 태스크 하나로 두고 activeForm만 단계에 맞게 갱신해도 된다.
- 모든 단계가 끝나면 3단계에서 **전체 통합 검수**를 한 번 더 한다 (단계별 검수는 그 단계 범위만 봤으므로,
  단계 간 연결·전체 동작을 최종 확인).

**호출 자체가 실패하면** (MCP 에러, 타임아웃 등): 새 `mcp__codex__codex` 세션으로 1회 재시도한다.
그래도 실패하면 해당 태스크를 `in_progress`로 남겨두고 사용자에게 상황을 알린다
(Codex MCP 연결 상태 확인 필요). 이때 사용자가 원하면 Claude가 직접 구현으로 전환할 수 있다.

**Codex 사용량이 다 된 경우 (quota exhausted) — Claude Sonnet 5 직접 구현으로 폴백**:
- 호출 전에 가능하면 `~/.claude/codex-status/codex-usage.json` 의
  `rate_limit.primary_window.used_percent` 를 확인해 100에 근접/도달했으면(예: 99 이상) 호출을
  시도하지 않고 바로 폴백한다.
- 호출 자체가 사용량/쿼터 관련 에러로 실패한 경우(에러 메시지에 "usage", "quota", "rate limit",
  "429" 등이 포함)도 동일하게 폴백 대상이다. 이 경우는 위 "호출 자체가 실패하면" 문단의 일반
  재시도 대상이 아니다 — 새 Codex 세션을 열어도 같은 이유로 또 실패하므로 재시도 없이 즉시
  폴백한다.
- 폴백 절차: 해당 `Codex 구현` 태스크는 그대로 두고, `Agent` 도구를 `model: "sonnet"` 으로 호출해
  (`subagent_type: "general-purpose"`) 1단계에서 작성한 구현 지시를 그대로 전달하고 Claude
  Sonnet 5가 직접 Edit/Write로 구현하게 한다. 태스크 설명에 "(Codex 사용량 소진 → Sonnet 5 폴백)"을
  덧붙여 기록한다.
- 폴백으로 구현된 부분도 3단계(결과 검수)를 동일하게 거친다. 재요청이 필요하면(4단계) 같은 폴백
  방식(Sonnet 5 Agent 재호출)으로 반복하고, 이때도 최대 3회 제한이 동일하게 적용된다.
- 병렬로 여러 세션을 돌리는 중 일부만 사용량이 소진됐다면, 그 세션들만 폴백하고 나머지는 계속
  Codex로 진행한다 (부분별로 독립적으로 판단).

### 3. 결과 검수 (Claude가 직접 수행)
`Claude 검수` 태스크를 `in_progress`로 표시한다.

**검수 강도 — 기본은 가벼운 검수다**: 테스트/빌드/타입체크 실행 + 요구사항 체크리스트 +
diff 훑어보기. 전체 diff를 한 줄씩 정독하지 않는다 (검수에 토큰을 다 쓰면 위임으로 아낀
의미가 없다). 단, 다음에 해당하는 부분만 골라서 정독한다: 보안 민감 코드(인증·권한·입력
처리·파일/DB 조작), 데이터 손상/유실 위험이 있는 부분, 외부에 공개되는 API/스키마 변경.

Codex 세션(들)이 끝나면 Claude가 직접:
- 0단계에서 기록한 기준선 대비 `git diff`로 (git이 아니면 기록해둔 파일들을 Read로) 실제 변경 내용을 확인한다.
  병렬로 여러 세션을 돌렸다면 전체 diff를 통합해서 본다.
- 가능하면 테스트/빌드/타입체크를 실행해 실제로 동작하는지 확인한다 (가장 우선하는 확인 수단).
- diff를 훑으며 확인: 요구사항 충족 여부, 명백한 버그/타입 오류, 기존 컨벤션 위반, 불필요한 범위 확장, 보안 이슈(OWASP top 10 등), 병렬 작업 간 충돌.
- 사용자에게 핵심 변경 부분을 코드로 보여주며 무엇을 확인했는지 간단히 알린다 (전체 diff를 다 붙이지 않는다).
- 문제가 없으면 `Claude 검수`를 `completed`로 표시하고 5단계로 넘어간다.

### 4. 문제 발견 시 — 자동 재요청
- **사소한 문제** (오타, import 누락, 한두 줄 수정으로 끝나는 것): Codex 왕복 없이 Claude가 직접 고치고
  검수를 이어간다. 무엇을 왜 직접 고쳤는지 마무리 보고에 포함한다.
- **그 외의 문제**: 사용자에게 묻지 않고, 해당 부분의 `Codex 구현` 태스크를 다시 `in_progress`로 돌린 뒤
  `mcp__codex__codex-reply`로 재요청한다:
  - `threadId`: 2단계에서 기록한 값 (문제가 생긴 부분의 세션)
  - `prompt`: 구체적으로 무엇이 잘못됐고 어떻게 고쳐야 하는지
- 재요청 후 다시 3단계(검수)로 돌아간다.
- 재요청은 **부분(세션)별로 최대 3회**까지 자동 반복한다. 3회 후에도 해결되지 않으면 반복을 멈추고,
  관련 태스크를 `in_progress` 상태로 남겨둔 채 사용자에게 현재 상태와 남은 문제를 설명한 뒤 어떻게 진행할지 묻는다.

### 5. 완료 보고
`Claude 검수`가 끝났으면 `마무리 보고`를 `in_progress`로 표시하고 사용자에게 간결히 요약한다:
- 무엇이 바뀌었는지 (파일 목록)
- 어떻게 테스트/확인했는지
- 남은 할 일이나 후속 제안 (있다면)

보고를 마치면 `마무리 보고`를 `completed`로 표시해 태스크 목록 전체를 완료 상태로 남긴다.

## 하지 않는 것
- 워크플로우에 들어간 뒤에는 Claude가 직접 Edit/Write로 기능 코드를 작성하지 않는다
  (예외: 게이트에서 "직접 구현"으로 판단한 작은 작업, 검수 중 발견한 한두 줄짜리 사소한 수정,
  그리고 Codex 사용량 소진 시의 Sonnet 5 폴백).
- 검수 없이 Codex 결과를 그대로 "완료"라고 보고하지 않는다.
- 사용자가 명시적으로 "네가 직접 짜줘"라고 하면 이 워크플로우를 건너뛰고 평소처럼 직접 구현한다.
- Codex MCP가 이 세션에 연결되어 있지 않으면 이 워크플로우를 쓸 수 없다 —
  사용자에게 알리고 직접 구현할지 확인한다.
